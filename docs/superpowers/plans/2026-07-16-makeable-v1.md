# Makeable v1 implementation plan

## Global constraints

- Build on branch `makeablev1` from `main`.
- The 11 PNGs in `Makeable figma/final-1440x1024/` are the visual source of truth; they must not be used as clickable page backgrounds.
- Preserve the approved visible elements and copy structure. Dynamic project, board, part, and repository data may replace illustrative values.
- Keep the vanilla HTML/CSS/JavaScript architecture and existing Node/Netlify API compatibility.
- Develop new behavior test-first and retain recoverable paths for AI, camera, serial, Arduino, and GitHub failures.
- Never expose or commit secrets, `.env`, user evidence, generated projects, `.DS_Store`, or `Makeable-v1-developer-handoff.zip`.
- Browser-hosted Makeable may plan and generate code, but flashing requires the local Makeable server.
- Handoff requires browser, accessibility, responsive, and visual verification plus `design-qa.md` with `final result: passed`.

## Task 1: Foundation and test harness

- Add the design handoff inputs needed at runtime, the supplied design tokens, self-hosted font files, and a vendored icon subset with licenses.
- Update `.gitignore` for generated evidence and redundant handoff artifacts.
- Add Node test, browser test, and accessibility tooling without introducing an application framework.
- Add a semantic Makeable document shell and split frontend modules under `src/makeable/`.
- Observe baseline tests failing before implementation and passing afterward.

## Task 2: State, routing, persistence, and secure configuration

- Implement the exact 11 routes, route-to-rail mapping, direct-load behavior, guarded progression, completed-screen revisits, and History API navigation.
- Persist project snapshots and image blobs in IndexedDB; use localStorage only for small configuration and legacy migration.
- Implement selective downstream invalidation for idea, photo, confirmed parts, wiring, firmware, tests, and publish state.
- Add `window.MAKEABLE_CONFIG` and retain `window.CIRCUIT_CODEX_CONFIG` only as an internal compatibility alias.
- Remove `deepgramApiKey` from public configuration, expose `hasDeepgramKey`, and add `POST /api/deepgram/token` locally and on Netlify.
- Add local and Netlify SPA fallback behavior for `/build/*`.

## Task 3: Describe, upload, review, and feasibility

- Build semantic, responsive Describe, Upload Parts, Review Parts, Ready, and Missing Parts screens matching the references.
- Make idea entry, example prompts, sketch attachment, voice subtitle, drag/drop, file picker, and camera capture operational.
- Use one normalized image pipeline for files and camera captures.
- Extend and normalize the AI plan contract for detected bounds, confidence, feasibility, missing parts, alternatives, and stable diagnostic metadata.
- Persist annotation selection, rename, deletion, bound adjustment, and low-confidence confirmation.
- Regenerate guide and firmware after parts confirmation; branch to Ready or Missing.
- Implement local missing-part acquisition state, targeted external searches, and inventory-compatible alternative projects without fake prices or checkout.

## Task 4: Assembly, code, flashing, and diagnostics

- Build responsive Assemble, Load Code, Automatic Test, and Manual Test screens matching the references.
- Show one persisted connection at a time with pins, wire color, explanation, quick check, Back, numbered progress, and completion.
- Use honest animated annotations on the user photo for instructional media.
- Preserve simple/advanced firmware views, edit, copy, download, actual detected board name, Arduino CLI status, compile, Web Serial/esptool flash, erase, progress, cancellation, disconnect, and retry.
- Require successful flashing before automatic testing.
- Parse stable serial markers, run safe sequential diagnostics, infer power stability only from reset/brownout evidence, stop resources safely, and link failures to assembly steps.
- Combine the requested real-world action, latest camera evidence, recent serial output, and OpenAI evaluation for manual testing, with repair and retry.

## Task 5: Publish, export, and success

- Build responsive Publish and Success screens matching the references.
- Preserve repository validation and public/private selection with server-side PAT configuration.
- Generate the visible artifact set: root README, build guide, `code/makeable.ino`, parts list, and test results.
- Preserve GitHub create/update recovery and SHA-based updates.
- Download the identical artifact set as a ZIP.
- Show the canonical repository URL, open-project action, Web Share with clipboard fallback, and Start Another Build that preserves environment configuration.

## Task 6: Integration, accessibility, responsive, and visual QA

- Browser-test ready and missing-parts paths with mocked AI, camera, serial, firmware, and GitHub services.
- Verify every visible control, keyboard focus order, live region, form label, reduced-motion behavior, contrast, and 44×44 touch target.
- Capture all 11 routes at 1440×1024 and the core flow at 834×1194 and 390×844.
- Compare each reference and implementation in a combined visual input, fix all P0/P1/P2 issues, and record each iteration in `design-qa.md`.
- Run the frontend-design and web-interface-guidelines audits, full automated tests, syntax/build checks, and an independent whole-branch code review.
- Leave the local server running and report the URL first at handoff.
